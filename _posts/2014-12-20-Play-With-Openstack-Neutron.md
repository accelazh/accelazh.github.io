Neutron consists of master node and agent node. On master nodes runs the controller service. Each agent node runs the agent service. Agent does the actual work to configure ovs, linux bridge, iptables, etc. Agent nodes are the data path. The master node communicates with agents by rpc on message queue.

In the past Neutron deal with driver heterogenety with plugins. A plugin includes code in controller part and agent part. There is a lot of duplicated code for each vendor to implement their own plugins. Later Neutron introduced [ML2 plugin](https://wiki.openstack.org/wiki/Neutron/ML2). ML2 devide concepts into MechanismDrivers(cisco, huawei, ovs, linux bridge, etc) and TypeDrivers(flat, vlan, gre, vxlan, etc). A vendor needs to implement one of them (see plugins/ml2/drivers). Other things are unified by ML2 itself. By now ML2 is the canonical plugin to use. Another great thing later added to ML2 is [L2 Pupulation](http://openstack.fr/r/meetup7/OpenStack-Neutron-l2-pop-Linux-VXLAN-Edouard-Thuleau.pdf). The plugin provide network connectivity and seperation for tenant and VMs.

About HA. Generally a failure of each agent services will only affact its local agent node. The iptabls rules, ovs rules, route tables are already written. So agent failure won't break existing traffic but only prevent adding new networks for a new VM. There are health monitors to periodically monitor the agents and restart.

Neutron also provide 'services', e.g. LBaaS, FWaaS, VPNaaS, which can be found in neutron/services. A services needs an agent running on a agent node. Take LBaaS as an example, a neutron-lbaas-agent process is running on each agent ndoe. There can be multiple lbaas agents. A scheduler is used to shard the load. Essentially neutron lbaas launches Haproxy processes to do the actual job. HA of the services agents works the same as plugin agents.

Neutron provides L3 services now, i.e. routers. You can use the l3 router, DVR (distributed virtual router), or provider router. L3 router launches a router agent on one node. All cross network traffic (east-west, south-north) goes through the router. If the router dies, traffic breaks. So, the l3 router has HA issue and scalability issue. DVR use multiple nodes to act as one router. Essentially it is built by iptables rules, ovs rules and linux route table. The traffic is sharded into different nodes. It is a promising but we have to wait for it to mature. Provider router means to use vendor proivded physical routers, such as cisco's.

Further code dive found out that, `cfg.CONF.ml2.mechanism_drivers` is empty, too. WHAT? neutron-server never loaded my config files! It seems neutron-server needs explicitly specify config files. This is different from nova. Gosh it tooks me totally 2 weeks to find out, to find the right place that goes wrong.
